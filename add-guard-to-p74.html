Wiom · Change Spec · v0.1 (draft) · PM deliverable

P74 guard — no retrigger once the customer's security fee is paid

Objective: once the customer has paid the security fee, the CSP has already made a significant, real investment to reach that point — Aadhar submitted, on-site work underway — and P74 should never be the reason that connection is taken away from them. Today P74 fires blind to this: it checks only whether the connection is still in PENDING_INSTALL, nothing about whether the fee is already paid, before reallocating the booking to a new CSP. This spec adds a guard — once the fee is paid, P74 no longer reallocates the connection — and removes the home-feed countdown timer that currently keeps counting down toward a reallocation that can no longer happen.

Author: Ashish Raj  ·  Reviewer: Engineering (pending)  ·  Tier: Medium — see note
Tier: Medium — the guard condition itself is simple, but CL (which owns P74) has zero visibility today into the fee-status field that TAS owns; building that cross-service visibility is a real architecture decision, not a config change. Status: Draft, under active audit — sections marked Pending below are placeholders, not omissions.
Rows: Required — every row filled or marked pending with a reason.

Quick Check

7-line triage.
What is changing?
Two things. (1) A new guard on P74's fire path. Before CL reallocates a connection via T3 on a P74 expiry, it checks whether the customer's security fee is PAID for the associated TAS execution candidate. If it is, P74 is a no-op — no reallocation. (2) The home-feed countdown disappears once paid. TAS already exposes a p74_deadline_at field that drives an existing countdown timer on the CSP/technician app's home feed; once the fee is PAID, TAS suppresses that field so the timer disappears — there's nothing left to count down to.
What triggered it?
PM's explicit principle: once the customer has paid, a significant investment has already been made by the CSP to reach that point, and the connection should not be automatically removed from the CSP/technician app because of a P74 retrigger. Verified via code trace that no existing guard checks this — P74's only check today is whether the connection is still in PENDING_INSTALL (SchedulingServiceImpl.loadInState).
Blast radius
CL (new guard logic inserted into the existing P74 fire path — SchedulingServiceImpl.processInstallTimeout and/or InboundEventProcessingServiceImpl.handleInstallationFailed, exact insertion point TBD; plus a new cross-service dependency on TAS's fee status, which CL has zero visibility into today) · TAS (owns the fee status already; needs to expose it to CL somehow, and separately needs to suppress its own p74DeadlineAt upstream field once paid — this second part is same-service, no new integration needed) · Gateway/App (no change needed — TaskCardResponse/HomeFeedService already just pass through whatever TAS provides).
Reversible?
The guard condition itself is easy to remove if wrong. The CL–TAS cross-service visibility mechanism, once built, is the less-reversible piece — worth getting the architecture right before committing (see Open Questions).
Money / PII / migration?
No new money movement — this only reads an existing fee-status field, never changes it. No PII. No migration: P74 timers already armed for in-flight connections are automatically covered by the new guard the next time they fire, since the check happens at fire time, not arm time.
OS impact / sign-off?
Pending — see A8. CL OS addition (new pre-condition on T3) · TAS OS clarification (fee status becomes a cross-service-relevant field, and the queue-card-upstream suppression is a new TAS-side behavior) · a Platform/Shared-Services OS question — CL lacking visibility into another service's business data.

Part A

WHAT — The Change

The PM's decisions: why, what the new flow looks like, what changes service-by-service.

Authored by the PM. Changes here trigger re-review.

— Objective WHAT

The broader objective is getting the connection activated on time — on or before the slot the CSP and customer already agreed on. P74 exists to make sure a connection doesn't sit forever with a CSP who isn't going to complete it: if the CSP doesn't move it out of PENDING_INSTALL in time, the booking is taken back and given to someone who might.

But once the customer has paid the security fee, the CSP has necessarily already done real work to get there — Aadhar has been submitted, on-site steps are underway (confirmed structurally: TAS's own step-gating logic requires Aadhar before the fee can even become payable). At that point, P74 pulling the connection away doesn't protect the customer from a stalled install — it punishes a CSP who has already invested real effort, for a delay that may have nothing to do with them.

This spec adds a guard: once the fee is paid, P74 stops being a threat to that connection at all — no reallocation — and the home-feed timer that currently counts down toward that reallocation disappears, since there's nothing left to count down to.

A2. Why This Change WHAT

What is expected: Observable trigger, who is affected, cost of leaving it.
QuestionAnswer
What's wrong with the current behaviour?P74's fire path checks exactly one thing before acting — is the connection still in PENDING_INSTALL (loadInState). It has no idea whether the customer has already paid the security fee. A CSP who has done the most (submitted Aadhar, gotten the customer to pay, is actively on-site) is treated identically to a CSP who has done nothing at all — both are subject to the exact same 72h clock, and both can be reallocated to a new CSP via T3.
Who experiences it?CSPs who have already invested real effort risk losing the booking — reassigned to a different CSP who starts from scratch — purely because a blind timer expired, not because of anything the CSP did wrong. Customers who have already paid could see their install handed to a brand-new CSP with no context, for a delay that isn't really abandonment. CSPs watching the home feed see a countdown ticking toward a reallocation that, once they've collected payment, should no longer be a real threat — creating confusion or unnecessary anxiety about a booking that's actually safe.
Cost of leaving it as-is?This is a live, structural risk, not a hypothetical — nothing in the current code prevents a paid, invested connection from being pulled via P74. It's presumably rare in practice only because most connections that reach fee payment also complete quickly, not because there's any actual protection against it.

A5. The Delta WHAT

What is expected: One row per atomic change: what changes → before → after → why.
What changesBeforeAfterWhy
New guard on P74's fire path — blocks T3 (reallocation) SchedulingServiceImpl.processInstallTimeout() only checks loadInState() (still PENDING_INSTALL) before raising InstallationFailed. handleInstallationFailed() unconditionally transitions to T3 — no business-condition check exists anywhere in this path today. Before P74 is allowed to act (exact insertion point TBD — see Open Questions), CL checks whether the customer's security fee is PAID for the connection's associated TAS execution candidate. If PAID, P74 is a graceful no-op: T3 does not fire, per explicit PM decision that a paid connection should not be automatically reassigned to a different CSP. Matches the base principle directly: once the fee is paid, real investment has already been made, and P74 should never be the reason that investment is voided.
New cross-service visibility — CL needs to know TAS's securityFeeStatus Confirmed via code search: zero references to security-fee tracking anywhere in csp-connection-lifecycle-service. CL has no field, no event subscription, no API call today that would tell it this status. Genuinely open architecture question, not decided here. Options include: (a) TAS publishes an outbox event on fee-paid that CL subscribes to and caches on its own connection entity; (b) CL makes a synchronous API call to TAS at guard-check time; (c) some other mechanism. See Open Questions in B5 — this is the single biggest unknown in this spec. CL cannot check a condition it has no way to observe. This is new integration work, not a config tweak, and deserves a deliberate architecture decision rather than an assumed default.
Home-feed countdown timer disappears once the fee is paid TasQueueCardUpstream.p74DeadlineAt is always populated while the connection is in PENDING_INSTALL, regardless of fee status — the existing home-feed timer has no awareness of payment either. TAS suppresses (nulls) p74DeadlineAt in its own queue-card upstream response once securityFeeStatus == PAID for the associated candidate. TAS already owns both the fee status and the field that feeds the gateway — this is a same-service change, unlike the guard above, and needs no new cross-service dependency. A countdown timer counting toward a removal that can no longer happen is actively misleading. Once paid, there's nothing left to warn the CSP about via this mechanism.
Feature flag for the guard — open question, not committed No existing flag infrastructure covers P74 at all today. This guard plausibly deserves its own flag — e.g. P74_SECURITY_FEE_GUARD_ENABLED — since it changes T3 behavior for every in-flight connection. Not committed here; flagged as an Open Question for PM/Engineering to confirm. If the guard has an unforeseen bug (e.g. mis-detecting fee status), an independent kill switch would let it be disabled without touching P74's core retry logic.

B5. Acceptance Criteria — Verification Layer HOW

What is expected: Given / When / Then blocks covering business logic and edge cases. Negative checks first-class on every block.

Business logic

LOGIC-1: Fee already PAID when P74 expires — guard blocks T3 (reallocation)

  Given a connection is in PENDING_INSTALL
  And   the associated TAS execution candidate's securityFeeStatus is PAID
  When  the P74 EventBridge timer fires
  Then  the new guard detects PAID status and blocks the pipeline
  And   InstallationFailed is never raised (or is raised but ignored downstream —
        exact insertion point TBD, see Open Questions)
  And   T3 does not fire — the connection stays exactly as it is,
        with the CSP still assigned

  Verify:
    • CL DB — connection current_state unchanged, still PENDING_INSTALL
    • CL DB — no new connection_event_history row for this P74 expiry
    • D&A — confirm no retry-intake event was ever received for this connection
LOGIC-2: Fee NOT paid when P74 expires — existing behavior is unchanged (regression check)

  Given a connection is in PENDING_INSTALL
  And   the associated TAS execution candidate's securityFeeStatus is PENDING or NOT_APPLICABLE
  When  the P74 EventBridge timer fires
  Then  the guard finds the fee is not PAID and does not block anything
  And   InstallationFailed is raised exactly as it is today
  And   T3 fires per the existing, unmodified logic

  Verify:
    • Identical to today's live P74 behavior — this is a regression check, not new behavior
    • Negative — confirm the guard adds no measurable delay to the fire path for the
      common (not-yet-paid) case, since this runs on every single P74 expiry going forward
LOGIC-3: Home-feed countdown disappears once the fee is paid

  Given a connection's home feed currently shows a P74 countdown (p74_deadline_at is set)
  When  CustomerSecurityFeePaid is processed for the associated candidate
  Then  TAS's queue-card upstream response for this candidate no longer includes
        p74DeadlineAt (or includes it as null)
  And   the home feed no longer shows a countdown for this task

  Verify:
    • Gateway API response — TaskCardResponse.p74_deadline_at is null/absent post-payment
    • Negative — confirm this doesn't affect any other task card field or any other
      connection's countdown
    • Negative — confirm a connection whose fee is still PENDING keeps showing its
      countdown exactly as before (no unintended suppression)
LOGIC-4: Race condition — fee paid at nearly the same moment P74 fires

  Given a connection is in PENDING_INSTALL, fee status is PENDING
  And   CustomerSecurityFeePaid and the P74 EventBridge callback arrive within
        a short window of each other
  When  the exact ordering is not guaranteed
  Then  behavior is currently undefined — this is a genuine open edge case,
        not asserted here (see Open Questions)

  Verify:
    • Not yet specified — needs an engineering decision on acceptable behavior for this
      race (e.g. best-effort/rare-and-acceptable vs. a reconciliation step) before this
      block can be completed
LOGIC-5: Already-armed P74 timers need no backfill

  Given a connection entered PENDING_INSTALL, and its P74 timer was armed,
        before this guard was deployed
  And   its fee is later marked PAID before the timer fires
  When  the already-armed timer fires after this guard is live
  Then  the guard still applies — it checks fee status at fire time, not at arm time
  And   no migration or backfill sweep is needed for in-flight connections

  Verify:
    • Confirm the guard reads current fee status live at fire time, not a value cached
      when the timer was armed
    • Negative — confirm this holds even for timers armed weeks before this guard shipped

Open questions for PM / Engineering (no silent defaults)

QuestionBest guess (needs confirmation)
How should CL learn TAS's securityFeeStatus — an outbox event CL subscribes to and caches, a synchronous API call at guard-check time, or something else?Not decided; this is the single biggest open item in this spec. An outbox-subscribed, locally-cached field avoids a synchronous cross-service call on every P74 fire (better for latency/availability), but introduces eventual-consistency risk (CL's cached copy could be briefly stale). A synchronous call is simpler and always current, but adds a hard runtime dependency on TAS being up whenever P74 fires. Needs an architecture review, not a PM guess.
Exact insertion point for the guard — before InstallationFailed is raised at all (in processInstallTimeout), or after, inside handleInstallationFailed?Leaning toward blocking earlier, in processInstallTimeout, so no failure event is raised at all for a paid connection — cleaner audit trail (nothing to explain away) than raising-then-ignoring. Not confirmed with engineering.
Does this guard need its own feature flag, e.g. P74_SECURITY_FEE_GUARD_ENABLED?Leaning yes, given this affects T3 for every in-flight connection. Not committed.
Race condition (LOGIC-4): fee paid at nearly the same moment P74 fires — is best-effort/rare-and-acceptable behavior fine, or does this need a reconciliation step (e.g. a compensating action if P74 fired just before the fee was marked paid)?Not decided. Given this is a genuinely rare timing window, leaning toward accepting best-effort behavior unless engineering flags a reason not to — but not asserted as a decision here.
Should there be a new reason code or audit-log entry marking "P74 checked but guarded due to paid fee," for traceability?Not decided. Worth considering so the audit trail can distinguish "P74 never fired" from "P74 fired but was silently guarded" — but adds a new field/value that isn't currently specified.

Out of scope

  • Designing or building the CL–TAS cross-service visibility mechanism itself — this spec states the requirement and flags it for an architecture review (A8), but does not design that integration here.
  • Any change to how the security fee is collected or how securityFeeStatus gets set — this spec is a read-only consumer of that existing field.
  • The race-condition handling in LOGIC-4 — flagged as an open question, not designed or committed to a specific resolution here.
