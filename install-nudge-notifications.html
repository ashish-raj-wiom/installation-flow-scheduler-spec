Installation nudges — Push Notification coverage

Objective: five install-flow conditions today only surface to the CSP via the in-app Attention Model (the Updates section) — which the CSP only sees if they happen to open the app. This spec adds a Push Notification for each of the five, so the CSP is proactively told, not just passively shown. This is a business-requirement spec only — use cases and target state, not implementation detail.

Author: Ashish Raj  ·  Reviewer: Engineering (pending)  ·  Tier: Light
Tier: Light — comms-only, no state changes, no reallocation logic. Status: Draft, under active audit.
Rows: Required — every row filled or marked pending with a reason.

Quick Check

7-line triage.
What is changing?
Five install-flow use cases (below) get a Push Notification via CleverTap. The Attention Model (Updates section) already covers 4 of the 5 today; the 5th needs an Attention Model update added too. No state changes, no reallocation — this is nudge/notification coverage only.
What triggered it?
Business requirement: CSPs are missing time-sensitive install steps because today's only signal (the Updates section) is passive — visible only if the CSP opens the app. A push closes that gap.
Blast radius
CSP app (Attention Model / Updates section — one new entry, 4 already exist) · CleverTap (5 new Push Notification campaigns, built and owned on the CleverTap side once the triggering event is available).
Reversible?
Yes — each is an independent push; any one can be turned off without touching the others or any state logic.

Part A

WHAT — The Change

The PM's decisions: why, what the new flow looks like, what changes service-by-service.

Authored by the PM. Changes here trigger re-review.

— Objective WHAT

Five points in the install flow are time-sensitive enough that the CSP should be told proactively, not left to discover them by opening the app. Today, the Attention Model (Updates section) already covers four of these five — but it's a passive surface. None of the five currently send a Push Notification.

This spec's scope is narrow and deliberate: it covers only these five nudges. It does not cover, and does not change, the separate reallocation/retrigger mechanism (RT) that moves a booking to a new CSP after a longer timeout — that is a different, already-published spec. Nothing here changes any install state.

A2. Why This Change WHAT

What is expected: Observable trigger, who is affected, cost of leaving it.
QuestionAnswer
What's wrong with the current behaviour?For 4 of the 5 use cases, the only signal today is an Attention Model entry in the Updates section — the CSP has to open the app to see it. For the 5th (Arrived at Site, no fee paid), there isn't even that. None of the 5 send a push.
Who experiences it?CSPs who don't happen to open the app during the window that matters — a slot that should've been proposed, a technician that should've been assigned, an OTP that should've been collected — miss the signal entirely until it's noticed some other way (or not at all).
Cost of leaving it as-is?Time-sensitive install steps slip further than they need to, purely because the CSP wasn't told proactively.

CSP backend's job (per use case): make sure the condition is checked and, when met, an event is sent that CleverTap can act on. B5. Acceptance Criteria — Verification Layer HOW

What is expected: Given / When / Then per use case, business level only.
LOGIC-1: Slot Not Proposed within 1 hr

  Given a candidate is assigned to a CSP and no slot has been proposed
  When  1 hour passes with no slot proposed
  Then  the Attention Model update fires (unchanged)
  And   a Push Notification is sent to the CSP (new)
LOGIC-2: Customer Confirmed but technician not assigned within 1 hr

  Given the customer has confirmed the slot and no technician is assigned
  When  1 hour passes with no technician assigned
  Then  the Attention Model update fires (unchanged)
  And   a Push Notification is sent to the CSP (new)
LOGIC-3: Technician Assigned but work not started on Slot day

  Given a technician is assigned and today is the slot day
  When  work has not started on the slot day
  Then  the Attention Model update fires (unchanged)
  And   a Push Notification is sent to the CSP (new)
LOGIC-4: Arrived at Site but no Fee Paid within 2 hrs

  Given the CSP has marked arrival and the fee is not yet paid
  When  2 hours pass with the fee still not paid
  Then  an Attention Model update fires (new)
  And   a Push Notification is sent to the CSP (new)
LOGIC-5: Awaiting OTP if Happy Code is not entered within 4 hrs

  Given the install is awaiting the customer's Happy Code / OTP
  When  4 hours pass with the code not entered
  Then  the Attention Model update fires (unchanged)
  And   a Push Notification is sent to the CSP (new)
LOGIC-6: Cancelled / declined / failed task sends no nudge (regression check, applies to all 5)

  Given a task would otherwise hit one of the 5 nudge conditions above
  And   the task is cancelled, declined, or failed before that window elapses
  When  the nudge's window would have elapsed
  Then  no Attention Model update fires for it
  And   no Push Notification is sent for it
  And   this holds even if the cancellation lands right before the nudge's window ends
