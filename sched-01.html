Wiom · Change Spec · v0.2 (draft) · PM deliverable · SCHED-01

Per-stage scheduler framework — full implementation of all TAS install schedulers

P74 today is one flat 72h timer anchored on CL's PENDING_INSTALL — a state that spans all 9 pre-arrival and 6 post-arrival TAS candidate states without distinguishing between them. It is also anchored differently than the OS specifies (see A2). This spec introduces a per-stage scheduler model — Service, Trigger State, ETA, Action States, Action, Feature Flag — and is the full implementation across every TAS-owned trigger state, not a single pilot: seven Trigger States, each independently flagged, replacing the one blind clock with the right objective at each stage. The RT mechanism itself is not new — P41 and P74/T3 already implement the full reroute pipeline; this spec's engineering ask is correctly-scoped triggers into that existing pipeline plus per-scheduler kill switches, not new reroute machinery.

Quick Check

7-line triage. Filled early this time — will be re-verified once B5/WR are complete.
What is changing?
Three coordinated changes. (1) A named scheduler model — every timer is now defined as (Service, Trigger State, ETA, Action States, Action, Feature Flag) instead of ad hoc. Action States resolves same-state-check vs. gate-check as a sizing question, not a binary choice: singleton for Nudge/System Intervention, multi-state span for RT — but only across states that lack their own scheduler; once every state in a chain has one, singleton is correct for RT too. (2) RT enabled at two states that were previously No-RT — AWAITING_TECHNICIAN_ASSIGNMENT and TECHNICIAN_ASSIGNED each get a new RT scheduler (singleton Action States, since their neighbor state now has its own coverage), alongside ARRIVED_AT_SITE's RT (multi-state span, since its neighbors don't yet have their own). (3) A feature flag per scheduler — SCHED_<TRIGGER_STATE>_<ACTION>_ENABLED — so any one scheduler, new or already-live, can be disabled independently without touching the others. All 7 TAS-owned Trigger States are in scope: AWAITING_SLOT_PROPOSAL, AWAITING_CUSTOMER_SLOT_CONFIRMATION, AWAITING_TECHNICIAN_ASSIGNMENT, TECHNICIAN_ASSIGNED, ARRIVED_AT_SITE, INSTALLATION_IN_PROGRESS_POST_FEE, AWAITING_CUSTOMER_OTP.
What triggered it?
Live scheduler audit (2026-07-07) found P74 firing across candidates in AWAITING_SLOT_PROPOSAL through ARRIVED_AT_SITE indiscriminately. Checking the OS directly (connection-lifecycle-prd.yaml, ASM-CLO-03) found P74 is specified to anchor on the INSTALLATION_COMPLETE event, not on PENDING_INSTALL entry — the live code anchors at CL's T1 instead. This is a spec-vs-code deviation, not a deliberate design choice, and it's the root cause of why a same-state RT check at any single TAS state silently under-protects the journey once the candidate moves one hop forward — which is why this spec covers every TAS trigger state at once rather than one at a time.
Blast radius
TAS (new outbound trigger equivalent to ES_INSTALL_REPORTED_FAILED/INSTALLATION_FAILED for each new RT scheduler, plus a new feature-flag check on every existing and new dispatch path) · CL (new reason codes accepted on the existing T3 path — no new state, no new transition) · D&A (zero change — the existing P41/T3 reroute pipeline is reused verbatim for every new RT trigger, see WF). No DB migration on D&A or CL. TAS needs a new schedule table entry pattern (mirrors existing per-connection EventBridge schedules from BUG-02/GAP-06) plus a flag-lookup table or config source for the feature flags.
Reversible?
Yes, per-scheduler — that's the entire point of the feature flag. Any single new scheduler can be switched off without affecting the other six. CL's existing P74 keeps running underneath as a backstop until each new trigger is proven out, then P74 can be narrowed back to its OS-specified INSTALLATION_COMPLETE anchor (separate follow-up item, not in this spec's scope).
Money / PII / migration?
No / No / No. Going-forward only. No existing candidate rows touched.
OS impact / sign-off?
Pending — see A8. Likely a CL OS clarification (new reason-code values on an existing transition, not a new transition) plus a TAS/Installation Flow OS addition (new per-candidate schedule types + flag config). No D&A OS change — D&A's side of the pipeline is untouched.

A2. Why This Change WHAT

What is expected: Observable trigger, who is affected, cost of leaving it.
QuestionAnswer
What's wrong with the current behaviour?Three connected problems. (1) One clock, wrong granularity. P74 is anchored on CL's PENDING_INSTALL, a single bucket spanning every TAS state from AWAITING_SLOT_PROPOSAL through AWAITING_CUSTOMER_OTP. It cannot distinguish "stuck waiting on a technician" from "stuck at the customer's door" — the same 72h applies uniformly, and the same downstream action (RT) fires regardless of ground truth. (2) OS-vs-code deviation. The OS specifies P74 anchored on the INSTALLATION_COMPLETE event (CSP self-reports on-site work done) — a narrow window near the end of the journey. Live code anchors it at CL's T1 (REQUESTED→PENDING_INSTALL, i.e. CSP acceptance) instead — the very start. This is why P74 was observed firing against candidates still in early pre-arrival states in the 2026-07-07 audit. (3) Same-state RT checks silently under-protect. An RT scheduler anchored on a single state (e.g. ARRIVED_AT_SITE alone) goes silent the instant the candidate advances one hop — if the next state (INSTALLATION_IN_PROGRESS_PRE_FEE) has no scheduler of its own, the candidate is now completely unprotected for as long as it likes, with no RT and no Nudge.
Who experiences it?CSPs: can lose a booking to RT with no advance warning (no Nudge exists on ARRIVED_AT_SITE today) and, before this fix, that same booking would silently escape RT coverage entirely if it inches forward from ARRIVED_AT_SITE. Customers: installs stall with no visible recovery mechanism in the exact window (post-arrival, pre-fee) where the CSP is on-site but not progressing. Quality/Enforcement OS: the RT-triggered DeclinePattern(TIMEOUT) signal never fires for this window, so a CSP who repeatedly no-shows post-arrival never accumulates a quality record for it.
Cost of leaving it as-is?The gaps this spec closes — no RT coverage past AWAITING_TECHNICIAN_ASSIGNMENT, TECHNICIAN_ASSIGNED, or ARRIVED_AT_SITE/PRE_FEE/FEE_COLLECTION_PENDING — are real, currently-live blind spots, not hypothetical. Every day they're open, a stalled candidate at any of these points has no system-driven recovery path. Rolling out all seven schedulers together, each behind its own flag, means the whole journey gets consistent coverage on day one, with the ability to pull back any single one if it misbehaves in production.

A candidate stuck post-arrival relies entirely on CL's blind 72h-from-acceptance clock. New per-candidate schedule armed on entry to ARRIVED_AT_SITE. Span (not singleton) because its two neighbor states have no scheduler of their own.

Three new, distinct failure_subreason_code values, one per new RT trigger. failure_subreason_source=SYSTEM, confidence per existing convention. Recorded on the terminal candidate row and in CL's connection_event_history payload exactly like TIMEOUT_P74 is today. Preserves the audit trail's ability to distinguish which of the three new schedulers caused a given RT — required per this session's discussion on traceability (see B5, verification block).
New Nudge schedulers, Trigger States = TECHNICIAN_ASSIGNED and ARRIVED_AT_SITE TECHNICIAN_ASSIGNED's Nudge was proposed but unbuilt; ARRIVED_AT_SITE had no Nudge at all — no warning exists before RT removes the booking at either. Singleton Action States at each of their own Trigger States. Fires before the corresponding RT ETA (exact lead times TBD) to tell the CSP the booking is about to be reassigned. No state change. Feature flags SCHED_TECHNICIAN_ASSIGNED_NUDGE_ENABLED, SCHED_ARRIVED_AT_SITE_NUDGE_ENABLED. RT should never be the CSP's first signal that something is wrong — matches the pattern already live for other stages (e.g. SLOT_PROPOSAL_URGENT, TECHNICIAN_ASSIGNMENT_URGENT).

B5. Acceptance Criteria — Verification Layer HOW

What is expected: Given / When / Then blocks covering business logic and edge cases. Negative checks first-class on every block.

Business logic

LOGIC-1: RT fires while stuck in the Trigger State itself

  Given a candidate enters ARRIVED_AT_SITE at time T
  And   no step is submitted before T + ETA (x hrs, TBD)
  When  the RT schedule fires
  Then  TAS verifies current_state is still in {ARRIVED_AT_SITE, INSTALLATION_IN_PROGRESS_PRE_FEE, FEE_COLLECTION_PENDING}
  And   TAS emits the INSTALLATION_FAILED equivalent with failure_reason=TIMEOUT_ARRIVED_AT_SITE
  And   CL fires T3 (PENDING_INSTALL to REQUESTED) exactly as it does for TIMEOUT_P74 today
  And   D&A reroutes per the existing pipeline (cooldown, P195, re-route, re-arm)
  And   the old candidate's terminal state = CANCELLED_BY_UPSTREAM with failure_reason=TIMEOUT_ARRIVED_AT_SITE

  Verify:
