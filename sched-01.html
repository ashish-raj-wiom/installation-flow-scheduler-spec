Wiom · Change Spec · v0.9 (draft) · PM deliverable · SCHED-01

Per-stage scheduler framework — a common scheduling service, event-driven, for all TAS install schedulers

P74 today is one flat 72h timer anchored on CL's PENDING_INSTALL — a state that spans all 9 pre-arrival and 6 post-arrival TAS candidate states without distinguishing between them. It is also anchored differently than the OS specifies (see A2). This spec introduces a per-stage scheduler model — Service, Trigger State, ETA, Action States, Action, Feature Flag, Outbox Event — as the full implementation across every TAS-owned trigger state, not a single pilot. Three things changed since v0.3: (1) the scheduling engine itself is now scoped as a common capability shared across D&A, CL, and TAS, not TAS-only infrastructure — today each service runs its own independently-duplicated EventBridgeTaskScheduler, and this spec should stop that duplication rather than add a fourth copy. (2) every scheduler fire, any Action type, must publish a domain event through the owning service's existing outbox so other services can subscribe — RT already satisfies this by construction, but Nudge and System Intervention publish nothing today. (3) a Nudge's outbox event should sit alongside the CSP-facing Attention model (the per-candidate signal that highlights a task in the CSP app's list) — verified as a real, TAS-owned mechanism that already fires correctly for install today (all 3 live Nudges already update it), so v0.4's original claim that this integration needed building was wrong and is corrected here; the outbox event adds cross-service observability without touching the working attention path. The RT reroute mechanism itself is still not new — P41 and P74/T3 already implement the full pipeline.

Quick Check

7-line triage. Filled early this time — will be re-verified once B5/WR are complete.
What is changing?
Five coordinated changes. (1) A named scheduler model — every timer is now defined as (Service, Trigger State, ETA, Action States, Action, Feature Flag, Outbox Event) instead of ad hoc. Action States resolves same-state-check vs. gate-check as a sizing question: singleton for Nudge/System Intervention, multi-state span for RT — only across states that lack their own scheduler. (2) RT enabled at two states that were previously No-RT — AWAITING_TECHNICIAN_ASSIGNMENT and TECHNICIAN_ASSIGNED, alongside ARRIVED_AT_SITE's RT. (3) A feature flag scoped to RT schedulers only — SCHED_<TRIGGER_STATE>_RT_ENABLED — 3 of 7 Trigger States get one in this spec (the 3 new RT triggers). D&A's P41 is the 4th RT-carrying Trigger State but its flag retrofit is confirmed out of scope for this version — not tracked as a follow-up here either, per explicit PM decision. (4) The scheduling engine becomes a common service — one shared arm/check/fire capability that D&A, CL, and TAS all register against, replacing the 3 independently-duplicated EventBridgeTaskScheduler implementations that exist today (one per service, same class name, no shared library). (5) Every scheduler fire publishes an Outbox Event, any Action type — RT already does this by construction (its reroute trigger is itself outbox-published), but Nudge and System Intervention currently publish nothing anywhere; this is genuinely new required behavior for those two types, including a retrofit for the 3 already-live Nudge schedulers. Correction (verified 2026-07-08): Nudge's Attention-model integration is already live for install — every live Nudge already calls `AttentionServiceImpl.incrementThreshold()`, bumping `latestAttentionVersion`/`Type`/`At`/`Reason` synchronously, using the `ReminderType` name as the reason. The new Outbox Event is purely additive, for cross-service observability — it does not drive, replace, or fix the attention update, which already works correctly today.
What triggered it?
Live scheduler audit (2026-07-07) found P74 firing across candidates in AWAITING_SLOT_PROPOSAL through ARRIVED_AT_SITE indiscriminately. Checking the OS directly (connection-lifecycle-prd.yaml, ASM-CLO-03) found P74 is specified to anchor on the INSTALLATION_COMPLETE event, not on PENDING_INSTALL entry — the live code anchors at CL's T1 instead. This is a spec-vs-code deviation, not a deliberate design choice, and it's the root cause of why a same-state RT check at any single TAS state silently under-protects the journey once the candidate moves one hop forward. The common-service and outbox-event asks (v0.4) came from a direct instruction: this scheduling capability should not be TAS-only, and every fire — not just RT — needs to be observable outside the owning service, with Nudge specifically wired into the CSP's existing Attention model instead of a passive, invisible-to-other-systems reminder flag.
Blast radius
TAS (new outbound trigger for each new RT scheduler, a new outbox event type for every Nudge/System Intervention scheduler — new and the 3 already-live ones, plus a new feature-flag check on the 3 new RT dispatch paths only — D&A's P41 is explicitly not touched) · CL (new reason codes accepted on the existing T3 path — no new state, no new transition) · D&A (zero change to the reroute pipeline itself or to P41's flag status, see WF — but in scope for the common-scheduling-engine ask, since D&A's own EventBridgeTaskScheduler would eventually consume the shared engine too). No DB migration on D&A or CL. The common scheduling engine is the widest-blast-radius item in this spec: it touches how all three services arm and check schedules, not just TAS's new ones — see A8 for the platform-ownership question this raises.
Reversible?
Yes, per-RT-scheduler, via the feature flag. The Outbox Event requirement is not independently reversible per scheduler — it's a structural requirement of the model, not a toggle. The common-scheduling-engine migration is the least reversible piece: once D&A/CL/TAS schedules move onto a shared engine, unwinding back to 3 independent implementations would be real rework, not a flag flip — this is exactly why WR below flags it for careful sequencing rather than a single big-bang cutover.
Money / PII / migration?
No / No / No new candidate-row migration. The common-scheduling-engine build may require a data migration of in-flight schedules off the 3 existing per-service implementations — TBD, depends on the chosen architecture (see WR).
OS impact / sign-off?
Pending — see A8. CL OS clarification (new failure_reason values) · TAS/Installation Flow OS addition (new schedule types, flag config for the 3 new RT triggers only) · a new Platform/Shared-Services OS question this spec surfaces for the first time: who owns a scheduling capability that spans D&A, CL, and TAS?

A2. Why This Change WHAT

What is expected: Observable trigger, who is affected, cost of leaving it.
QuestionAnswer
What's wrong with the current behaviour?Five connected problems. (1) One clock, wrong granularity. P74 is anchored on CL's PENDING_INSTALL, a single bucket spanning every TAS state from AWAITING_SLOT_PROPOSAL through AWAITING_CUSTOMER_OTP. It cannot distinguish "stuck waiting on a technician" from "stuck at the customer's door" — the same 72h applies uniformly, and the same downstream action (RT) fires regardless of ground truth. (2) OS-vs-code deviation. The OS specifies P74 anchored on the INSTALLATION_COMPLETE event (CSP self-reports on-site work done) — a narrow window near the end of the journey. Live code anchors it at CL's T1 (REQUESTED→PENDING_INSTALL, i.e. CSP acceptance) instead — the very start. (3) Same-state RT checks silently under-protect. An RT scheduler anchored on a single state goes silent the instant the candidate advances one hop, if the next state has no scheduler of its own. (4) Scheduling infrastructure is duplicated three times. TAS, CL, and D&A each run their own independently-implemented EventBridgeTaskScheduler — same class name, no shared code — so every service that wants per-stage scheduling re-solves the same arm/check/fire problem from scratch. (5) Nudge is invisible outside TAS itself. Every live Nudge scheduler today (SLOT_PROPOSAL_URGENT, TECHNICIAN_ASSIGNMENT_URGENT, HAPPY_CODE_SUBMIT_URGENT) fires with no event published anywhere — confirmed by tracing the full call chain (onThresholdReminder → incrementThreshold → incrementAttention), which contains no eventPublisher/outbox call at all. Correction (verified 2026-07-08): the CSP's own task-list Attention signal is not missing for install — it already updates correctly on every one of these fires (latestAttentionVersion/Type/At/Reason, using the ReminderType name as the reason). The actual gap is narrower than first documented: nothing outside TAS can observe a Nudge firing, because the update happens entirely in-process with no outbox event — other services have no way to subscribe.
Who experiences it?CSPs: can lose a booking to RT with no advance warning. Nudges themselves already surface correctly on the CSP's task list today (attention signal updates as designed) — that part is not broken. Customers: installs stall with no visible recovery mechanism in the exact window (post-arrival, pre-fee) where the CSP is on-site but not progressing. Quality/Enforcement OS: the RT-triggered DeclinePattern(TIMEOUT) signal never fires for uncovered windows, so a CSP who repeatedly no-shows never accumulates a quality record for it. Other backend services: today have no way to know a scheduler fired at all unless it happens to be an RT (which incidentally routes through the outbox already) — a service that might want to react to a Nudge or System Intervention (e.g. ops tooling, analytics, a future retention feature) has nothing to subscribe to.
Cost of leaving it as-is?The RT coverage gaps are real, currently-live blind spots — every day they're open, a stalled candidate has no system-driven recovery path. Separately, every day the scheduling engine stays TAS-only, the next service that needs per-stage scheduling (D&A, CL, or a future one) either duplicates the pattern a fourth time or blocks on a rewrite later — the cost compounds the longer duplication is allowed to continue. And every day Nudge stays outbox-silent, no other service can react to a candidate being nudged — the CSP-facing side already works today (attention signal updates correctly), so this cost is about cross-service visibility, not CSP-facing UX.

Business logic

LOGIC-1: RT fires while stuck in the Trigger State itself

  Given a candidate enters ARRIVED_AT_SITE at time T
  And   no step is submitted before T + ETA (x hrs, TBD)
  When  the RT schedule fires
  Then  TAS verifies current_state is still in {ARRIVED_AT_SITE, INSTALLATION_IN_PROGRESS_PRE_FEE, FEE_COLLECTION_PENDING}
  And   TAS emits the INSTALLATION_FAILED equivalent with failure_reason=TIMEOUT_ARRIVED_AT_SITE
  And   CL fires T3 (PENDING_INSTALL to REQUESTED) exactly as it does for TIMEOUT_P74 today
  And   D&A reroutes per the existing pipeline (cooldown, P195, re-route, re-arm)
  And   the old candidate's terminal state = CANCELLED_BY_UPSTREAM with failure_reason=TIMEOUT_ARRIVED_AT_SITE,
        failure_subreason_code=CSP_NO_SHOW (reused, not a new value)

  Verify:
    • TAS DB — install_execution_candidates WHERE execution_candidate_id = <test>
        expect current_state = 'CANCELLED_BY_UPSTREAM', failure_reason = 'TIMEOUT_ARRIVED_AT_SITE',
        failure_subreason_code = 'CSP_NO_SHOW', failure_subreason_source = 'SYSTEM'
    • CL DB — connection_event_history WHERE connection_id = <test>
        expect one INSTALLATION_FAILED row, source_os = SYSTEM, resulting_state = 'REQUESTED'
    • D&A DB — connection_allocations WHERE connection_id = <test>
        expect a decline-cooldown entry for the old CSP, assignment_count incremented
    • Negative — no RT fires before T + ETA even if the candidate is visibly stalled
        (this scheduler must not fire early under any condition)
LOGIC-2: RT fires from a later Action State — the loophole this spec exists to close

  Given a candidate enters ARRIVED_AT_SITE at time T
  And   the CSP submits the SELFIE step, advancing to INSTALLATION_IN_PROGRESS_PRE_FEE at T + 10min
  And   no further progress happens before T + ETA
  When  the RT schedule (armed at T, on ARRIVED_AT_SITE entry) fires at T + ETA
  Then  TAS still checks current_state against the Action States SET, not against the original Trigger State alone
  And   INSTALLATION_IN_PROGRESS_PRE_FEE IS a member of the Action States set
  And   RT fires exactly as in LOGIC-1

  Verify:
    • Same as LOGIC-1, but confirm the candidate's failure_reported_at state (pre_withdrawal_state
      equivalent) = 'INSTALLATION_IN_PROGRESS_PRE_FEE', not 'ARRIVED_AT_SITE'
    • Negative — this is the case a same-state-only check would silently miss. Explicitly verify
      the OLD (naive) implementation would NOT have fired here, to prove the fix.

  Given a candidate enters ARRIVED_AT_SITE at time T

  Verify:
    • TAS DB — candidate current_state unchanged by this schedule's fire
    • No new outbox_record entries attributable to this scheduler for this candidate
    • This is the base-principle boundary (no RT once both sides have invested) enforced at the
