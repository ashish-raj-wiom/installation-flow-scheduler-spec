Wiom · Change Spec · v0.1 (draft) · PM deliverable · SCHED-01

Per-stage scheduler framework — a common scheduling service, event-driven, for all TAS install schedulers

P74 today is one flat 72h timer anchored on CL's PENDING_INSTALL — a state that spans all 9 pre-arrival and 6 post-arrival TAS candidate states without distinguishing between them. It is also anchored differently than the OS specifies (see A2). This spec introduces a per-stage scheduler model — Service, Trigger State, ETA, Action States, Action, Feature Flag, Outbox Event — as the full implementation across every TAS-owned trigger state, not a single pilot, built around four coordinated ideas: (1) the scheduling engine itself is scoped as a common capability shared across D&A, CL, and TAS, not TAS-only infrastructure — today each service runs its own independently-duplicated EventBridgeTaskScheduler, and this spec should stop that duplication rather than add a fourth copy. (2) every scheduler fire, any Action type, must publish a domain event through the owning service's existing outbox so other services can subscribe — RT already satisfies this by construction, but Nudge and System Intervention publish nothing today. (3) a Nudge's outbox event sits alongside the CSP-facing Attention model (the per-candidate signal that highlights a task in the CSP app's list) — verified as a real, TAS-owned mechanism that already fires correctly for install today (all 3 live Nudges already update it); the outbox event adds cross-service observability without touching the working attention path. (4) every ETA is a named, configurable parameter and projects through the existing Working Hours service rather than naive calendar time — both already true in production for some existing timers, verified via code trace, but inconsistently and mostly switched off. The RT reroute mechanism itself is still not new — P41 and P74/T3 already implement the full pipeline.

What is changing?
Six coordinated changes. (1) A named scheduler model — every timer is now defined as (Service, Trigger State, ETA, Action States, Action, Feature Flag, Outbox Event) instead of ad hoc. Action States resolves same-state-check vs. gate-check as a sizing question: singleton for Nudge/System Intervention, multi-state span for RT — only across states that lack their own scheduler. (2) RT enabled at two states that were previously No-RT — AWAITING_TECHNICIAN_ASSIGNMENT and TECHNICIAN_ASSIGNED, alongside ARRIVED_AT_SITE's RT. (3) A feature flag scoped to RT schedulers only — SCHED_<TRIGGER_STATE>_RT_ENABLED — 3 of 7 Trigger States get one in this spec (the 3 new RT triggers). D&A's P41 is the 4th RT-carrying Trigger State but its flag retrofit is confirmed out of scope for this version — not tracked as a follow-up here either, per explicit PM decision. (4) The scheduling engine becomes a common service — one shared arm/check/fire capability that D&A, CL, and TAS all register against, replacing the 3 independently-duplicated EventBridgeTaskScheduler implementations that exist today (one per service, same class name, no shared library). (5) Every scheduler fire publishes an Outbox Event, any Action type — RT already does this by construction (its reroute trigger is itself outbox-published), but Nudge and System Intervention currently publish nothing anywhere; this is genuinely new required behavior for those two types, including a retrofit for the 3 already-live Nudge schedulers. Separately, Nudge's Attention-model integration is already live for install — every live Nudge already calls `AttentionServiceImpl.incrementThreshold()`, bumping `latestAttentionVersion`/`Type`/`At`/`Reason` synchronously, using the `ReminderType` name as the reason. The new Outbox Event is purely additive, for cross-service observability — it does not drive, replace, or fix the attention update, which already works correctly today. (6) Every new ETA respects working hours. A real WorkingHoursService already exists to project a duration into an actual fire time inside a 09:00–21:00 IST window, but verified it's inconsistently wired and mostly off by default across every existing timer that could use it — this spec's 5 new ETAs default to using it, rather than adding a 4th instance of "supported but switched off."
What triggered it?
Live scheduler audit (2026-07-07) found P74 firing across candidates in AWAITING_SLOT_PROPOSAL through ARRIVED_AT_SITE indiscriminately. Checking the OS directly (connection-lifecycle-prd.yaml, ASM-CLO-03) found P74 is specified to anchor on the INSTALLATION_COMPLETE event, not on PENDING_INSTALL entry — the live code anchors at CL's T1 instead. This is a spec-vs-code deviation, not a deliberate design choice, and it's the root cause of why a same-state RT check at any single TAS state silently under-protects the journey once the candidate moves one hop forward. The common-service and outbox-event asks came from a direct instruction: this scheduling capability should not be TAS-only, and every fire — not just RT — needs to be observable outside the owning service, with Nudge sitting alongside the CSP's existing Attention model rather than only a passive, invisible-to-other-systems reminder flag.
Blast radius
TAS (new outbound trigger for each new RT scheduler, a new outbox event type for every Nudge/System Intervention scheduler — new and the 3 already-live ones, plus a new feature-flag check on the 3 new RT dispatch paths only — D&A's P41 is explicitly not touched) · CL (new reason codes accepted on the existing T3 path — no new state, no new transition) · D&A (zero change to the reroute pipeline itself or to P41's flag status, see WF — but in scope for the common-scheduling-engine ask, since D&A's own EventBridgeTaskScheduler would eventually consume the shared engine too). No DB migration on D&A or CL. The common scheduling engine is the widest-blast-radius item in this spec: it touches how all three services arm and check schedules, not just TAS's new ones — see A8 for the platform-ownership question this raises. All 5 new ETAs add a runtime dependency on csp-gateway-service's working-hours endpoint via TAS's existing WorkingHoursClient — an existing dependency, not a new one, but this spec is the first thing to actually rely on it working correctly by default rather than treating it as optional.
Reversible?
Yes, per-RT-scheduler, via the feature flag. The Outbox Event requirement is not independently reversible per scheduler — it's a structural requirement of the model, not a toggle. The common-scheduling-engine migration is the least reversible piece: once D&A/CL/TAS schedules move onto a shared engine, unwinding back to 3 independent implementations would be real rework, not a flag flip — this is exactly why WR below flags it for careful sequencing rather than a single big-bang cutover.
Money / PII / migration?
No / No / No new candidate-row migration. The common-scheduling-engine build may require a data migration of in-flight schedules off the 3 existing per-service implementations — TBD, depends on the chosen architecture (see WR).
OS impact / sign-off?
Pending — see A8. CL OS clarification (new failure_reason values) · TAS/Installation Flow OS addition (new schedule types, flag config for the 3 new RT triggers only) · a new Platform/Shared-Services OS question this spec surfaces for the first time: who owns a scheduling capability that spans D&A, CL, and TAS?

Part A

WHAT — The Change

The PM's decisions: why, what the new flow looks like, what changes service-by-service.

Authored by the PM. Changes here trigger re-review.

— Objective WHAT

The broader objective is simple: make sure the connection gets activated on time — on or before the slot both the CSP and the customer have already agreed on. Everything else in this spec is in service of that one outcome.

P74 exists for exactly this reason. It gives the CSP a bounded window to move the connection out of PENDING_INSTALL. If the CSP doesn't get there in time, the booking is taken back and handed to someone else who might actually be able to install it before the customer's patience — and the SLA — run out.

The problem is that P74, as a single flat clock, doesn't account for two very different situations that land on the exact same timer:

  • Customer-side delay in slot confirmation. When the customer, not the CSP, is the one holding things up, the CSP is not the blocker — but P74 doesn't know that, and keeps ticking against them anyway.
  • On-ground installation steps where the CSP has already made major investment. Once a technician has traveled to site and started the actual work, real time and effort has already gone in — but the booking still gets pulled the moment the same blind timeout expires, regardless of how close the CSP actually was to finishing.

This spec solves this by replacing the one blind clock with state-based schedulers — a timer per stage, each carrying the action that actually fits what's true at that stage — instead of one flat timeout applied uniformly to every situation.

A2. Why This Change WHAT

What is expected: Observable trigger, who is affected, cost of leaving it.
QuestionAnswer
What's wrong with the current behaviour?Seven connected problems. (1) One clock, wrong granularity. P74 is anchored on CL's PENDING_INSTALL, a single bucket spanning every TAS state from AWAITING_SLOT_PROPOSAL through AWAITING_CUSTOMER_OTP. It cannot distinguish "stuck waiting on a technician" from "stuck at the customer's door" — the same 72h applies uniformly, and the same downstream action (RT) fires regardless of ground truth. (2) OS-vs-code deviation. The OS specifies P74 anchored on the INSTALLATION_COMPLETE event (CSP self-reports on-site work done) — a narrow window near the end of the journey. Live code anchors it at CL's T1 (REQUESTED→PENDING_INSTALL, i.e. CSP acceptance) instead — the very start. (3) Same-state RT checks silently under-protect. An RT scheduler anchored on a single state goes silent the instant the candidate advances one hop, if the next state has no scheduler of its own. (4) Scheduling infrastructure is duplicated three times. TAS, CL, and D&A each run their own independently-implemented EventBridgeTaskScheduler — same class name, no shared code — so every service that wants per-stage scheduling re-solves the same arm/check/fire problem from scratch. (5) Nudge is invisible outside TAS itself. Every live Nudge scheduler today (SLOT_PROPOSAL_URGENT, TECHNICIAN_ASSIGNMENT_URGENT, HAPPY_CODE_SUBMIT_URGENT) fires with no event published anywhere — confirmed by tracing the full call chain (onThresholdReminder → incrementThreshold → incrementAttention), which contains no eventPublisher/outbox call at all. The CSP's own task-list Attention signal is not missing for install, though — it already updates correctly on every one of these fires (latestAttentionVersion/Type/At/Reason, using the ReminderType name as the reason). The actual gap is narrower: nothing outside TAS can observe a Nudge firing, because the update happens entirely in-process with no outbox event — other services have no way to subscribe. (6) ETAs mostly ignore working hours. A real WorkingHoursService exists to project a duration into an actual fire time inside a defined working window, but P41 and 2 of the 3 live TAS reminders have it wired behind a flag that defaults off, the 3rd live reminder has no flag at all and always computes naively, and CL has no client for it whatsoever — so in practice almost every timer in this system today can fire at any hour, including the middle of the night, despite the mechanism to prevent that already existing. (7) A Nudge before RT is a one-time flag, not a live warning. Even once the 2 new Nudge schedulers exist, they only add a single Updates-section entry — nothing shows the CSP a persistent, ticking indication of exactly how much time is left before the booking is removed. A CSP who doesn't happen to open the Updates feed at the right moment gets no ongoing signal at all.
Who experiences it?CSPs: can lose a booking to RT with no advance warning, and can have that RT — or a Nudge warning — fire at 2am with the working-hours protection sitting unused right next to it. Nudges themselves already surface correctly on the CSP's task list today (attention signal updates as designed) — that part is not broken. Customers: installs stall with no visible recovery mechanism in the exact window (post-arrival, pre-fee) where the CSP is on-site but not progressing. Quality/Enforcement OS: the RT-triggered DeclinePattern(TIMEOUT) signal never fires for uncovered windows, so a CSP who repeatedly no-shows never accumulates a quality record for it. Other backend services: today have no way to know a scheduler fired at all unless it happens to be an RT (which incidentally routes through the outbox already) — a service that might want to react to a Nudge or System Intervention (e.g. ops tooling, analytics, a future retention feature) has nothing to subscribe to.
Cost of leaving it as-is?The RT coverage gaps are real, currently-live blind spots — every day they're open, a stalled candidate has no system-driven recovery path. Separately, every day the scheduling engine stays TAS-only, the next service that needs per-stage scheduling (D&A, CL, or a future one) either duplicates the pattern a fourth time or blocks on a rewrite later — the cost compounds the longer duplication is allowed to continue. Every day Nudge stays outbox-silent, no other service can react to a candidate being nudged — the CSP-facing side already works today (attention signal updates correctly), so this cost is about cross-service visibility, not CSP-facing UX. And every day the new ETAs default to the same off-by-default working-hours gating as the existing timers, a CSP can get an RT-reroute notice or a Nudge warning at an hour they have no realistic chance to act on it — a cost that compounds specifically because the fix (the working-hours service) already exists and is simply not switched on.

LOGIC-1: RT fires while stuck in the Trigger State itself

  Given a candidate enters ARRIVED_AT_SITE at time T
  And   no step is submitted before T + INSTALL_ARRIVED_AT_SITE_RT_ETA (numeric value TBD)
  When  the RT schedule fires
  Then  TAS verifies current_state is still in {ARRIVED_AT_SITE, INSTALLATION_IN_PROGRESS_PRE_FEE, FEE_COLLECTION_PENDING}
  And   TAS emits the INSTALLATION_FAILED equivalent with failure_reason=TIMEOUT_ARRIVED_AT_SITE
  And   CL fires T3 (PENDING_INSTALL to REQUESTED) exactly as it does for TIMEOUT_P74 today
  And   D&A reroutes per the existing pipeline (cooldown, P195, re-route, re-arm)
  And   the old candidate's terminal state = CANCELLED_BY_UPSTREAM with failure_reason=TIMEOUT_ARRIVED_AT_SITE,
        failure_subreason_code=CSP_NO_SHOW (reused, not a new value)

  Verify:
    • TAS DB — install_execution_candidates WHERE execution_candidate_id = <test>
        expect current_state = 'CANCELLED_BY_UPSTREAM', failure_reason = 'TIMEOUT_ARRIVED_AT_SITE',
        failure_subreason_code = 'CSP_NO_SHOW', failure_subreason_source = 'SYSTEM'
    • CL DB — connection_event_history WHERE connection_id = <test>
        expect one INSTALLATION_FAILED row, source_os = SYSTEM, resulting_state = 'REQUESTED'
    • D&A DB — connection_allocations WHERE connection_id = <test>
        expect a decline-cooldown entry for the old CSP, assignment_count incremented
    • Negative — no RT fires before T + ETA even if the candidate is visibly stalled
        (this scheduler must not fire early under any condition)
LOGIC-2: RT fires from a later Action State — the loophole this spec exists to close

  Given a candidate enters ARRIVED_AT_SITE at time T
  And   the CSP submits the SELFIE step, advancing to INSTALLATION_IN_PROGRESS_PRE_FEE at T + 10min
  And   no further progress happens before T + INSTALL_ARRIVED_AT_SITE_RT_ETA
  When  the RT schedule (armed at T, on ARRIVED_AT_SITE entry) fires at T + INSTALL_ARRIVED_AT_SITE_RT_ETA
  Then  TAS still checks current_state against the Action States SET, not against the original Trigger State alone
  And   INSTALLATION_IN_PROGRESS_PRE_FEE IS a member of the Action States set
  And   RT fires exactly as in LOGIC-1

  Verify:
    • Same as LOGIC-1, but confirm the candidate's failure_reported_at state (pre_withdrawal_state
      equivalent) = 'INSTALLATION_IN_PROGRESS_PRE_FEE', not 'ARRIVED_AT_SITE'
    • Negative — this is the case a same-state-only check would silently miss. Explicitly verify
      the OLD (naive) implementation would NOT have fired here, to prove the fix.

    • This is the base-principle boundary (no RT once both sides have invested) enforced at the

  Verify:

    • Negative — if the same candidate entered ARRIVED_AT_SITE at T = 10:00 IST (fully inside the
      window), projectedDue and naive T.plus(2h) must agree (both = 12:00 IST same day) — the
      working-hours projection must be a no-op when the whole window fits inside working hours,
      not a source of drift for the common case
    • Negative — confirm this is exercised for all 5 new ETAs in this spec (3 RT + 2 Nudge), not
      just ARRIVED_AT_SITE's RT — a partial rollout would leave some new schedulers still firing
      at any hour while others don't
    • Negative — per WR row 15, both existing WorkingHoursClient copies fail open (fall back to

    • Negative — this must be built for both new Nudge schedulers (TECHNICIAN_ASSIGNED,
