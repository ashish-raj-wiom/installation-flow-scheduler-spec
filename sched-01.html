Wiom · Change Spec · v0.1 (draft) · PM deliverable · SCHED-01

Per-stage scheduler framework for installation flow — pilot item: Arrived-at-Site RT

P74 today is one flat 72h timer anchored on CL's PENDING_INSTALL — a state that spans all 9 pre-arrival and 6 post-arrival TAS candidate states without distinguishing between them. It is also anchored differently than the OS specifies (see A2). This spec introduces a per-stage scheduler model — Service, Trigger State, ETA, Action States, Action — and pilots it end-to-end on the first concrete gap this audit found: ARRIVED_AT_SITE has no RT coverage today, and a same-state-only check would leave INSTALLATION_IN_PROGRESS_PRE_FEE and FEE_COLLECTION_PENDING completely unprotected the moment the CSP takes one step forward. The RT mechanism itself is not new — P41 and P74/T3 already implement the full reroute pipeline; this spec's engineering ask is a correctly-scoped trigger into that existing pipeline, not new reroute machinery.

Quick Check

7-line triage. Filled early this time — will be re-verified once B5/WR are complete.
What is changing?
Two coordinated changes. (1) A named scheduler model — every timer is now defined as (Service, Trigger State, ETA, Action States, Action) instead of ad hoc. Action States is the field that resolves same-state-check vs. gate-check as a single design question: size it to a singleton for Nudge/System Intervention (message must match the exact stuck state) or to a multi-state span for RT (RT doesn't care which pre-investment state the task is stuck in, only that it hasn't crossed the fee-payment line). (2) A new TAS-owned RT scheduler at ARRIVED_AT_SITE, Action States = {ARRIVED_AT_SITE, INSTALLATION_IN_PROGRESS_PRE_FEE, FEE_COLLECTION_PENDING}, that triggers the existing INSTALLATION_FAILED → T3 → ClStateChanged → reroute pipeline (today only reachable via CL's P74) with a new reason code TIMEOUT_ARRIVED_AT_SITE. A companion Nudge scheduler (singleton Action States = {ARRIVED_AT_SITE}) warns the CSP before the RT window closes.
What triggered it?
Live scheduler audit (2026-07-07) found P74 firing across candidates in AWAITING_SLOT_PROPOSAL through ARRIVED_AT_SITE indiscriminately. Checking the OS directly (connection-lifecycle-prd.yaml, ASM-CLO-03) found P74 is specified to anchor on the INSTALLATION_COMPLETE event, not on PENDING_INSTALL entry — the live code anchors at CL's T1 instead. This is a spec-vs-code deviation, not a deliberate design choice, and it's the root cause of why a same-state RT check at any single TAS state silently under-protects the journey once the candidate moves one hop forward.
Blast radius
TAS (new outbound trigger equivalent to ES_INSTALL_REPORTED_FAILED/INSTALLATION_FAILED, fired from a new per-candidate schedule instead of waiting on CL's P74) · CL (new reason code accepted on the existing T3 path — no new state, no new transition) · D&A (zero change — the existing P41/T3 reroute pipeline is reused verbatim, see WF). No DB migration on D&A or CL. TAS needs a new schedule table entry pattern (mirrors existing per-connection EventBridge schedules from BUG-02/GAP-06).
Reversible?
Yes — the new TAS-side trigger is additive; CL's existing P74 keeps running underneath as a backstop until this is proven out, then P74 can be narrowed back to its OS-specified INSTALLATION_COMPLETE anchor (separate follow-up item, not in this spec's scope).
Money / PII / migration?
No / No / No. Going-forward only. No existing candidate rows touched.
OS impact / sign-off?
Pending — see A8. Likely a CL OS clarification (new reason-code value on an existing transition, not a new transition) plus a TAS/Installation Flow OS addition (new per-candidate schedule type). No D&A OS change — D&A's side of the pipeline is untouched.

A2. Why This Change WHAT

What is expected: Observable trigger, who is affected, cost of leaving it.
QuestionAnswer
What's wrong with the current behaviour?Three connected problems. (1) One clock, wrong granularity. P74 is anchored on CL's PENDING_INSTALL, a single bucket spanning every TAS state from AWAITING_SLOT_PROPOSAL through AWAITING_CUSTOMER_OTP. It cannot distinguish "stuck waiting on a technician" from "stuck at the customer's door" — the same 72h applies uniformly, and the same downstream action (RT) fires regardless of ground truth. (2) OS-vs-code deviation. The OS specifies P74 anchored on the INSTALLATION_COMPLETE event (CSP self-reports on-site work done) — a narrow window near the end of the journey. Live code anchors it at CL's T1 (REQUESTED→PENDING_INSTALL, i.e. CSP acceptance) instead — the very start. This is why P74 was observed firing against candidates still in early pre-arrival states in the 2026-07-07 audit. (3) Same-state RT checks silently under-protect. An RT scheduler anchored on a single state (e.g. ARRIVED_AT_SITE alone) goes silent the instant the candidate advances one hop — if the next state (INSTALLATION_IN_PROGRESS_PRE_FEE) has no scheduler of its own, the candidate is now completely unprotected for as long as it likes, with no RT and no Nudge.
Who experiences it?CSPs: can lose a booking to RT with no advance warning (no Nudge exists on ARRIVED_AT_SITE today) and, before this fix, that same booking would silently escape RT coverage entirely if it inches forward from ARRIVED_AT_SITE. Customers: installs stall with no visible recovery mechanism in the exact window (post-arrival, pre-fee) where the CSP is on-site but not progressing. Quality/Enforcement OS: the RT-triggered DeclinePattern(TIMEOUT) signal never fires for this window, so a CSP who repeatedly no-shows post-arrival never accumulates a quality record for it.
Cost of leaving it as-is?The gap this spec closes (Arrived-at-Site through Fee-Collection-Pending having no independent RT/Nudge coverage) is a real, currently-live blind spot — not hypothetical. Every day it's open, a stalled on-site install has no system-driven recovery path.

A5. The Delta WHAT

What is expected: One row per atomic change: what changes → before → after → why.
What changesBeforeAfterWhy
Scheduler definition model Ad hoc — P74/P41/reminders each defined independently, no shared vocabulary. "Same-state vs. gate-check" was an unresolved, per-timer judgment call. Every scheduler is (Service, Trigger State, ETA, Action States, Action). Action States is the field that resolves same-state-vs-gate-check: singleton for Nudge/System Intervention, multi-state span for RT (up to the fee-payment boundary). One vocabulary for all timers going forward; the sizing rule is explicit and auditable per scheduler instead of implicit per-implementation.
New TAS-owned RT trigger, Trigger State = ARRIVED_AT_SITE No RT scheduler exists at or after ARRIVED_AT_SITE independent of CL's mis-anchored P74. A candidate stuck post-arrival relies entirely on CL's blind 72h-from-acceptance clock. New per-candidate schedule armed on entry to ARRIVED_AT_SITE. Action States = {ARRIVED_AT_SITE, INSTALLATION_IN_PROGRESS_PRE_FEE, FEE_COLLECTION_PENDING} — fires RT if the candidate is still in any of these three states at ETA (x hrs, TBD — see Open Questions in B5). On fire, TAS emits the equivalent of today's CSP-self-reported INSTALLATION_FAILED, with a new reason code TIMEOUT_ARRIVED_AT_SITE, into the existing, unmodified CL T3 → D&A reroute pipeline. Closes the exact gap A2 describes with the minimum new surface: reuses the entire existing reroute pipeline (see WF and the RT implementation checklist), only adds a correctly-scoped trigger.
New reason code TIMEOUT_ARRIVED_AT_SITE The only system-timeout reason code on this path is TIMEOUT_P74, which conflates every possible stuck-state into one label — impossible to tell from the audit trail which scheduler actually fired. New, distinct failure_subreason_code value. failure_subreason_source=SYSTEM, confidence per existing convention. Recorded on the terminal candidate row and in CL's connection_event_history payload exactly like TIMEOUT_P74 is today. Preserves the audit trail's ability to distinguish which scheduler caused a given RT — required per this session's discussion on traceability (see B5, verification block).
New Nudge scheduler, Trigger State = ARRIVED_AT_SITE No warning exists before RT removes the booking. Singleton Action States = {ARRIVED_AT_SITE}. Fires before the RT ETA (exact lead time TBD) to tell the CSP the booking is about to be reassigned. No state change. RT should never be the CSP's first signal that something is wrong — matches the pattern already live for other stages (e.g. SLOT_PROPOSAL_URGENT, TECHNICIAN_ASSIGNMENT_URGENT).

B5. Acceptance Criteria — Verification Layer HOW

What is expected: Given / When / Then blocks covering business logic and edge cases. Negative checks first-class on every block.

Business logic

LOGIC-1: RT fires while stuck in the Trigger State itself

  Given a candidate enters ARRIVED_AT_SITE at time T
  And   no step is submitted before T + ETA (x hrs, TBD)
  When  the RT schedule fires
  Then  TAS verifies current_state is still in {ARRIVED_AT_SITE, INSTALLATION_IN_PROGRESS_PRE_FEE, FEE_COLLECTION_PENDING}
  And   TAS emits the INSTALLATION_FAILED equivalent with failure_reason=TIMEOUT_ARRIVED_AT_SITE
  And   CL fires T3 (PENDING_INSTALL to REQUESTED) exactly as it does for TIMEOUT_P74 today
  And   D&A reroutes per the existing pipeline (cooldown, P195, re-route, re-arm)
  And   the old candidate's terminal state = CANCELLED_BY_UPSTREAM with failure_reason=TIMEOUT_ARRIVED_AT_SITE

  Verify:
    • TAS DB — install_execution_candidates WHERE execution_candidate_id = <test>
        expect current_state = 'CANCELLED_BY_UPSTREAM', failure_reason = 'TIMEOUT_ARRIVED_AT_SITE'
    • CL DB — connection_event_history WHERE connection_id = <test>
        expect one INSTALLATION_FAILED row, source_os = SYSTEM, resulting_state = 'REQUESTED'
    • D&A DB — connection_allocations WHERE connection_id = <test>
        expect a decline-cooldown entry for the old CSP, assignment_count incremented
    • Negative — no RT fires before T + ETA even if the candidate is visibly stalled
        (this scheduler must not fire early under any condition)
LOGIC-2: RT fires from a later Action State — the loophole this spec exists to close

  Given a candidate enters ARRIVED_AT_SITE at time T
  And   the CSP submits the SELFIE step, advancing to INSTALLATION_IN_PROGRESS_PRE_FEE at T + 10min
  And   no further progress happens before T + ETA
  When  the RT schedule (armed at T, on ARRIVED_AT_SITE entry) fires at T + ETA
  Then  TAS still checks current_state against the Action States SET, not against the original Trigger State alone
  And   INSTALLATION_IN_PROGRESS_PRE_FEE IS a member of the Action States set
  And   RT fires exactly as in LOGIC-1

  Verify:
    • Same as LOGIC-1, but confirm the candidate's failure_reported_at state (pre_withdrawal_state
      equivalent) = 'INSTALLATION_IN_PROGRESS_PRE_FEE', not 'ARRIVED_AT_SITE'
    • Negative — this is the case a same-state-only check would silently miss. Explicitly verify
      the OLD (naive) implementation would NOT have fired here, to prove the fix.
LOGIC-3: No RT once genuinely progressed past the Action States set

  Given a candidate enters ARRIVED_AT_SITE at time T
  And   the CSP progresses all the way to INSTALLATION_IN_PROGRESS_POST_FEE (fee paid) before T + ETA
  When  the RT schedule fires at T + ETA
  Then  TAS checks current_state, finds it is NOT in {ARRIVED_AT_SITE, PRE_FEE, FEE_COLLECTION_PENDING}
  And   the scheduler is a graceful no-op — no cancellation, no reroute, no event emitted

  Verify:
    • TAS DB — candidate current_state unchanged by this schedule's fire
    • No new outbox_record entries attributable to this scheduler for this candidate
    • This is the base-principle boundary (no RT once both sides have invested) enforced at the
      implementation level, not just documented as a policy
